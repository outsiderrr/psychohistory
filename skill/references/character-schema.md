# Character Card Schema & Validation Rules (v1.1 polymorphic)

> This document defines the strict data standard for all character cards used in Psychohistory.
> Every card — whether from the official library, generated by Nuwa CLI, or pasted by a user in chat — MUST pass this validation.
>
> **v1.1 change**: the schema is now **polymorphic**. A single schema file supports three distinct card types via a `card_type` discriminator and JSON Schema `oneOf` conditional validation. Previously the schema only modeled personal entities and had no structural home for organizations or collectives.

---

## 1. Design Overview

### Three card types

| `card_type` | Use case | Primary structural fields (beyond the common core) |
|---|---|---|
| `personal-entity` | Individual decision-makers (heads of state, CEOs, founders, judges) | `mental_models`, `decision_heuristics`, `concession_triggers`, `red_lines` |
| `organization-entity` | Governments, central banks, corporations, armed forces, international bodies | The above + `decision_structure`, `organizational_inertia`, `communication_style`, `key_dependencies`, `historical_precedents` |
| `collective` | Voter bases, market participants, demographic groups — forces without a single decision-maker | `composition`, `core_interests`, `current_disposition`, `sensitivity_map`, `influence_pathway`, `observable_state` |

### Why polymorphic rather than four separate schemas

The three card types share a significant common core (`card_version`, `card_type`, `agent_id`, `name`, `source`, `honesty_boundaries`, optional `alternative_frameworks`). Splitting into multiple files would require duplicating these definitions and maintaining them in parallel.

Polymorphic schema (one file, `oneOf` + `card_type` discriminator) avoids this duplication, lets standard JSON Schema Draft 7+ validators handle validation directly, and keeps the three types visually side-by-side for comparison and editing.

### Not a character card: relationships

Relationship definitions (`loyalty`, `influence`, `alliance`, `rivalry`, `dependency`) are structural links **between** character cards, not character cards themselves. Their schema lives in `schema.md §3 Agent Relationship`, separate from this file.

### Structure of the schema below

- **Common fields** (required by all card types): `card_version`, `card_type`, `agent_id`, `name`, `source`, `honesty_boundaries`
- **Common optional field**: `alternative_frameworks` (theory validation hook)
- **Personal + Organization shared fields**: `role`, `affiliation`, `mental_models`, `decision_heuristics`, `concession_triggers`, `red_lines`, `expression_dna`, `values_hierarchy`, `known_biases`
- **Organization-only fields**: `decision_structure`, `organizational_inertia`, `communication_style`, `key_dependencies`, `historical_precedents`
- **Collective-only fields**: `description`, `composition`, `core_interests`, `current_disposition`, `sensitivity_map`, `influence_pathway`, `observable_state`, `historical_behavior`

The `oneOf` block at the bottom enforces per-type required fields.

## 3. Validation Rules (Human-Readable)

### 3.1 Common requirements (all card types)

| Field | Rule |
|---|---|
| `card_version` | Must be `"1.0"` |
| `card_type` | One of: `personal-entity`, `organization-entity`, `collective` |
| `agent_id` | Lowercase alphanumeric + hyphens only (`^[a-z0-9-]+$`) |
| `name` | Non-empty English string |
| `source.type` | One of: `official`, `nuwa-skill`, `user`, `system-derived` |
| `source.data_cutoff` | Complete `YYYY-MM-DD` format (no partial dates) |
| `honesty_boundaries` | At least 1 item; must include the type-specific disclaimer listed in §3.2 / §3.3 / §3.4 |

### 3.2 Personal-entity additional requirements

| Field | Rule |
|---|---|
| `role` | Non-empty |
| `mental_models` | 2-10 items, each with `id` / `name` / `description` (≥20 chars) |
| `decision_heuristics` | 2-12 items |
| `concession_triggers` | ≥1 item |
| `red_lines` | ≥1 item |
| `honesty_boundaries` **must include** | (1) Source material bias disclaimer (2) Private deliberation invisibility disclaimer (3) Data staleness disclaimer |

### 3.3 Organization-entity additional requirements

| Field | Rule |
|---|---|
| `role` | Non-empty |
| `mental_models` | 2-10 items (organization-level) |
| `decision_heuristics` | 2-12 items |
| `concession_triggers` | ≥1 item |
| `red_lines` | ≥1 item |
| `decision_structure` | Present with `type` (one of: `autocratic` / `committee` / `consensus` / `hybrid`) |
| `organizational_inertia` | Present with `current_trajectory` + `change_resistance` |
| `honesty_boundaries` **must include** | Factional divergence disclaimer (e.g. "This card reflects the dominant faction X at data_cutoff; non-dominant factions would diverge on fields A, B, C") |

### 3.4 Collective additional requirements

| Field | Rule |
|---|---|
| `description` | Non-empty (used in place of `role`) |
| `core_interests` | ≥2 items, each with `priority`, `description` (≥20 chars), optionally `driver` |
| `current_disposition` | Present with `issue` + `stance` |
| `sensitivity_map` | ≥2 items, each with `event_type` + `sensitivity` |
| `influence_pathway` | Present with `target_agent` + `mechanism` |
| `observable_state` | ≥1 item, each with `state_id` + `description` |
| `honesty_boundaries` **must include** | Internal heterogeneity disclaimer (e.g. "This profile reflects mean disposition; sub-group X diverges on issue Y") |
| **Must NOT contain** | `mental_models`, `decision_heuristics`, `concession_triggers`, `red_lines` — collectives without a unified decision-maker have no unified cognition. These fields are enforced absent by convention, not by schema constraint |

### 3.5 Quality checks (all types)

| Check | Pass Criteria | Fail Signal |
|---|---|---|
| Description depth | Each `description` ≥ 20 characters | One-word descriptions |
| ID format | `mm-01`, `dh-01`, `ct-01`, `alt-xxx` patterns | Random strings |
| Honesty boundary specificity | Lists concrete limitations | Only says "not the real person/org/group" |
| Type-specific disclaimer | Present in `honesty_boundaries` per §3.2/§3.3/§3.4 | Missing |
| Evidence traceability (phase-based prompts) | Each conclusion maps to a `references.md` section | Conclusions with no source |
| Strength grading | Each `mental_model` / `decision_heuristic` / `concession_trigger` / `core_interest` / `sensitivity_map` entry has a `strength` field | Missing strength = implicit `high` (bad) |
| Internal consistency | `mental_models` / `core_interests` do not directly contradict each other without explanation | Logical inconsistencies |

### 3.6 Validation error messages

```
⚠️ Character card validation failed for [name] (card_type: [type]):
- [FIELD]: [specific error]
- [FIELD]: [specific error]
Please fix the above issues and resubmit.
```
